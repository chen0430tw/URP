# 系统能力概述

> 日期：2026-03-28

本文档用通俗语言说明 URP 项目两个核心组件各自能做什么。

---

## 一、iqvw64e.sys / kdmapper — 内核通道

iqvw64e.sys 是英特尔网卡诊断驱动，kdmapper 把它当成一个**可操控的内核访问通道**来利用。它本身是真实的 Windows 内核驱动，只是被当工具用。

**需要管理员权限才能运行。**

### 能做的事

**内核层权限**
- 加载未签名驱动：绕过 DSE（Driver Signature Enforcement），把任意 `.sys` 文件塞进内核运行，不需要微软签名
- 任意内核内存读写：可以访问任意内核虚拟地址
- 执行内核代码：把 shellcode 写进内核池内存后直接调用，在 Ring 0 执行

**信息获取**
- `GetKernelModuleAddress`：找 `ntoskrnl.exe`、`win32k.sys` 等模块基址
- `GetKernelModuleExport`：拿任意内核函数地址（相当于内核版的 `GetProcAddress`）

**反取证**
- `ClearMmUnloadedDrivers`：从内核的驱动卸载记录里抹掉痕迹
- `ClearPiDDBCacheTable`：从 PatchGuard 用来检测手动映射驱动的缓存里删掉条目

### 实际应用场景

- 游戏外挂（读取游戏进程内核结构、绕过反作弊）
- 安全研究 / 内核调试（在没有 WDK 调试器的情况下探测内核）
- 红队工具（提权后的持久化、EDR 绕过）
- 驱动开发测试（快速加载未签名测试驱动，不用开 TestSigning 模式）

### 限制

- 需要管理员权限才能加载驱动
- Windows 11 24H2+ 默认启用 Vulnerable Driver Blocklist，会直接拦截 iqvw64e.sys
- PatchGuard 监控内核结构，有被 BSOD 的风险

## 二、URP Runtime — 异构算力调度运行时

URP 是一个**把多种计算资源统一管理起来的执行引擎**，核心理念是：

> **万物皆为可调度的算力节点**

### 它做的事

把计算任务表示成一张图（IRGraph），然后自动决定哪段代码在哪个节点上运行（CPU / GPU / 远程机器），通过数据包传递中间结果，最后合并输出。

```
定义计算图 → 融合/分区优化 → 策略调度到节点 → 本地/远程执行 → 合并结果
```

### 调度考虑的因素

| 因素 | 说明 |
|------|------|
| 标签匹配 | 任务要求 `gpu`，只有带 `gpu` 标签的节点才能接 |
| 区域偏好 | 优先调度到同区域节点，减少跨区延迟 |
| 算力容量 | 同等条件下选容量更大的节点 |
| 惯性亲和 | 上次处理过相同数据的节点优先，利用缓存局部性 |
| 路由开销 | 同机器走内存环形队列（快），跨机器走网络包（慢） |

### kdmapper 在里面的角色

kdmapper 是 URP 的**硬件扩展工具**。

如果你有一块 FPGA、定制 ASIC 或者没有官方驱动的加速卡，通过 kdmapper 把自己写的驱动装进内核，然后 URP 就能把那块硬件当成一个算力节点调度进来。

```
自定义硬件
    │
    └─ kdmapper 加载无签名驱动 → 暴露硬件能力给用户空间
                                        │
                                        └─ 封装为 URP Node (tag="fpga")
                                                   │
                                                   └─ IRGraph 中 required_tag=fpga 的块
                                                      自动被调度到这个节点执行
```

简单说：**URP 管调度，kdmapper 管接硬件。**
