# 系统能力概述

> 日期：2026-03-28（v1.0 更新）

本文档用通俗语言说明 URP 项目各核心组件能做什么。

---

## 一、URP Runtime — 异构算力调度运行时

URP 是一个**把多种计算资源统一管理起来的执行引擎**，核心理念是：

> **万物皆为可调度的算力节点**

把计算任务表示成一张图（IRGraph），然后自动决定哪段代码在哪个节点上运行（CPU / GPU / USB 外设），通过数据包传递中间结果，最后合并输出。

```
定义计算图 → 融合/分区优化 → 策略调度到节点 → 本地/远程执行 → 合并结果
```

### 调度考虑的因素

| 因素 | 说明 |
|------|------|
| 标签匹配 | 任务要求 `gpu`，只有带 `gpu` 标签的节点才能接 |
| 区域偏好 | 优先调度到同区域节点，减少跨区延迟 |
| 算力容量 | 同等条件下选容量更大的节点 |
| 惯性亲和 | 上次处理过相同数据的节点优先，利用缓存局部性 |
| 路由开销 | 同机器走内存环形队列（快），跨机器走网络包（慢） |
| ET-WCN 冷却 | 熵温度感知的分区绑定，在高熵状态下优化任务分配 |

## 二、执行后端

### CPU 执行器（默认）

`BlockExecutor` 在本机 CPU 上执行所有 40+ 条 URX opcode，包括：
- 整型算术 / 逻辑 / 位运算
- 浮点运算（F64，含三角函数、指数、幂次）
- 类型转换、控制流

### GPU 执行器（feature="gpu"）

`WgpuExecutor` 通过 wgpu + WGSL 着色器在 GPU 上执行向量运算。
`JitExecutor` 配合 JIT 编译器，将 IRGraph 动态编译为 WGSL 后在 GPU 上运行。

适用场景：大批量数值计算（N=1K~1M 的向量加法、sqrt、L2 范数等）。

### USB 执行器（feature="usb"）

`UsbExecutor` 通过 USB CDC 串口协议与外部计算设备通信，将 URX opcode 打包发送，等待设备返回结果。

**这是真正的异构卸载**：设备（如运行 urp-pico 固件的 Raspberry Pi Pico）在自己的 CPU 上执行计算，结果通过 USB 返回主机。

```
IRBlock(UAdd) → UsbExecutor → [USB 帧] → RP2040 固件 exec_op() → [USB 帧] → PayloadValue::I64(42)
```

`UsbDiscovery` 自动扫描所有串口，通过 HELLO 握手识别已知 VID/PID（RP2040/STM32/Pi）的 URP 设备，并返回 `DeviceInfo`（名称、能力、吞吐量）。

### 重要：普通 U 盘不能作为计算节点

URP USB 执行器要求设备以 **CDC ACM（串口类）** 呈现，在 Windows 下显示为 COMx 串口。

普通 U 盘（如 SanDisk、Kingston DataTraveler 等）是 **USB MSC（Mass Storage Class）** 设备，在系统中挂载为盘符（E:\、G:\ 等），**不会出现在串口列表中**，`UsbDiscovery::scan()` 无法找到它们。

| | 普通 U 盘 | URP 计算节点 |
|--|--|--|
| USB 协议类 | MSC（Mass Storage） | CDC ACM（Serial） |
| Windows 呈现 | 盘符（E:\、G:\） | 串口（COM3、COM4） |
| 主控可编程 | 固件锁死，只做存储 | 用户可烧录自定义固件 |
| 能否执行 URX opcode | 否 | 是 |

**可用设备示例**：Raspberry Pi Pico（RP2040）烧录 `firmware/urp-pico/` 后插入，即出现为 CDC 串口，可被 `UsbDiscovery` 自动识别。

## 三、USB 线路协议

所有 URP 设备使用统一帧格式：

```
[0xA5][LEN_LO][LEN_HI][PAYLOAD...][CRC8]
```

- CRC8：poly=0x31，MSB-first，init=0x00（与固件 `crc8()` 完全一致）
- 协议细节见 README.md "USB 线路协议" 章节

## 四、RP2040 参考固件（firmware/urp-pico/）

`firmware/urp-pico/` 是基于 TinyUSB 的 RP2040（Raspberry Pi Pico）固件，实现完整的 URP 设备端逻辑：

- USB CDC ACM，无需额外驱动（Windows/Linux/macOS 即插即用）
- VID=0x2E8A / PID=0x000A，由 `UsbDiscovery` 自动识别
- 序列号从 Flash UID 自动生成
- 支持全部 40+ 条 URX opcode，含 USelect（3 输入）、HELLO 握手
- RX 状态机逐字节解帧，保证鲁棒性

拖拽 .uf2 到 Pico 即可使用，无需 IDE 或调试器。

## 五、iqvw64e.sys / kdmapper — 内核通道

iqvw64e.sys 是英特尔网卡诊断驱动，kdmapper 把它当成一个**可操控的内核访问通道**来利用。

**需要管理员权限才能运行。**

### 能做的事

**内核层权限**
- 加载未签名驱动：绕过 DSE（Driver Signature Enforcement），把任意 `.sys` 文件塞进内核运行
- 任意内核内存读写
- 执行内核代码（Ring 0）

**信息获取**
- `GetKernelModuleAddress`：找内核模块基址
- `GetKernelModuleExport`：拿任意内核函数地址

**反取证**
- `ClearMmUnloadedDrivers`：从内核驱动卸载记录里抹掉痕迹
- `ClearPiDDBCacheTable`：从 PatchGuard 缓存里删掉条目

### 在 URP 中的角色

kdmapper 是 URP 的**内核硬件扩展工具**。通过 kdmapper 把自定义驱动装进内核，URP 就能把未官方支持的硬件（FPGA、定制 ASIC 等）封装为算力节点参与调度。

```
自定义硬件
    └─ kdmapper 加载无签名驱动 → 暴露硬件能力给用户空间
                                      └─ 封装为 URP Node (tag="fpga")
                                                 └─ IRGraph 自动调度到该节点
```

### 限制

- 需要管理员权限
- Windows 11 24H2+ 默认启用 Vulnerable Driver Blocklist，会拦截 iqvw64e.sys
- PatchGuard 监控内核结构，有被 BSOD 的风险

## 六、当前各后端能力对比

| 后端 | 真实硬件执行 | 所需条件 | 状态 |
|------|------------|---------|------|
| CPU | 本机 CPU | 无 | ✅ 默认启用 |
| GPU (wgpu) | 本机 GPU | feature="gpu"，支持 Vulkan/Metal/DX12 | ✅ 已实现 |
| USB (CDC) | 外部设备 CPU | feature="usb" + URP 固件设备 | ✅ 已实现 |
| KDMapper | 内核 Ring 0 | 管理员权限 + feature="kdmapper" | ✅ FFI 已实现 |
| Remote | 远程节点 | 网络可达 | ✅ 已实现 |
