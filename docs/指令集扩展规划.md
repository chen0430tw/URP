# URP 指令集扩展规划

参考 MIPS32 设计，结合 URP 现有架构（IRBlock / PayloadValue / HardwareExecutor）制定。

---

## 一、现状

| Opcode | 类型 | 说明 |
|--------|------|------|
| `UConstI64(i64)` | 常量 | 返回 i64 字面量 |
| `UConstStr(String)` | 常量 | 返回字符串字面量 |
| `UAdd` | 算术 | i64 加法（已有 CPU + GPU 实现） |
| `UConcat` | 字符串 | 两值拼接为字符串 |

MergeMode（结果合并策略，非指令）：`List`、`Sum`、`Concat`、`ReduceMax`

---

## 二、MIPS 分类对照与选取原则

MIPS 将指令分为 7 大类：算术、逻辑、移位、内存、数据移动、分支跳转、陷阱。
URP 是**数据流图执行模型**，没有 PC、没有寄存器文件，因此部分 MIPS 概念需要映射：

| MIPS 概念 | URP 对应 |
|-----------|---------|
| 寄存器 | IRBlock 的 `inputs`（按名引用） |
| 立即数 | `UConstI64` / `UConstStr` block |
| 内存 load/store | 不适用（URP 通过 PayloadValue 传值，无地址空间） |
| 分支/跳转 | 不适用（控制流由 DAG 拓扑决定，非条件跳转） |
| HI/LO 特殊寄存器 | 不适用（URP 无隐式输出寄存器） |
| 陷阱指令 | 可用 `UAssert` 实现前置条件检查 |
| 浮点协处理器 | 对应 GPU executor（可扩展 f64 类型） |

**跳过的 MIPS 类别：** 内存访问（load/store）、分支延迟槽、PC 相对跳转。
**重点借鉴的 MIPS 类别：** 算术、逻辑、移位、比较（SLT 系列）、数据移动（条件移动 MOVZ/MOVN）。

## 三、扩展指令集规划

### 第一批（核心算术与比较）

优先级最高，覆盖最常用运算，可直接映射 GPU shader。

| Opcode | 输入 | 输出 | 对标 MIPS | 说明 |
|--------|------|------|-----------|------|
| `USub` | i64, i64 | i64 | `SUBU` | 减法，无溢出 trap |
| `UMul` | i64, i64 | i64 | `MUL`（R2） | 乘法，取低 64 位 |
| `UDiv` | i64, i64 | i64 | `DIVU` + `MFLO` | 整除（商） |
| `URem` | i64, i64 | i64 | `DIVU` + `MFHI` | 取余数 |
| `UCmpEq` | i64, i64 | i64 (0/1) | `BEQ` 的条件部分 | 相等返回 1，否则 0 |
| `UCmpLt` | i64, i64 | i64 (0/1) | `SLTU` | 小于返回 1 |
| `UCmpLe` | i64, i64 | i64 (0/1) | `SLT` + 取反 | 小于等于 |

> **设计说明**：MIPS 的 DIV 同时产生商（LO）和余数（HI），需要 MFLO/MFHI 两条指令取出。
> URP 没有隐式输出，所以分成独立的 `UDiv`（商）和 `URem`（余数）两个 opcode，
> 各自产生一个 PayloadValue 输出——更符合数据流模型。

---

### 第二批（逻辑与移位）

对标 MIPS 逻辑类 + 移位类，适合位操作场景。

| Opcode | 输入 | 输出 | 对标 MIPS | 说明 |
|--------|------|------|-----------|------|
| `UAnd` | i64, i64 | i64 | `AND` | 按位与 |
| `UOr` | i64, i64 | i64 | `OR` | 按位或 |
| `UXor` | i64, i64 | i64 | `XOR` | 按位异或 |
| `UNot` | i64 | i64 | `NOR $r, $0` | 按位取反 |
| `UShl` | i64, i64 | i64 | `SLLV` | 左移（移位量来自第二个输入） |
| `UShr` | i64, i64 | i64 | `SRLV` | 逻辑右移（零填充） |
| `UShra` | i64, i64 | i64 | `SRAV` | 算术右移（符号位填充） |

> **设计说明**：MIPS 把移位量固定（`SLL`）和可变（`SLLV`）分成两条指令，
> 是因为 R-type 的 shamt 字段只有 5 位，装不了寄存器编号。
> URP 无此限制——`UShl(a, b)` 中 b 可以是任意 i64 block 的输出，直接统一为可变版本。

---

### 第三批（类型转换与字符串）

补全 PayloadValue 两种类型（i64 / Str）之间的互转。

| Opcode | 输入 | 输出 | 对标 MIPS | 说明 |
|--------|------|------|-----------|------|
| `UI64ToStr` | i64 | Str | `cvt.w` 系列 | 整数转十进制字符串 |
| `UStrToI64` | Str | i64 | — | 解析字符串为整数，失败 panic |
| `UStrLen` | Str | i64 | — | 字符串长度 |
| `UStrSlice` | Str, i64, i64 | Str | — | 子串 [start, end) |
| `UStrSplit` | Str, Str | Str (多输出) | — | 按分隔符拆分（MergeMode::List） |

> **设计说明**：MIPS 通过 `mtc1/mfc1` 在整数和浮点寄存器之间传值。
> URP 类比：`UI64ToStr` / `UStrToI64` 是两种 PayloadValue 类型之间的"通道"指令。

---

### 第四批（条件选择与聚合）

对标 MIPS 的 `MOVZ`/`MOVN`（条件移动）和 MergeMode 聚合。

| Opcode | 输入 | 输出 | 对标 MIPS | 说明 |
|--------|------|------|-----------|------|
| `USelect` | i64 (cond), any, any | any | `MOVZ`/`MOVN` | cond≠0 选第二输入，否则选第三输入 |
| `UMin` | i64, i64 | i64 | — | 两值取小 |
| `UMax` | i64, i64 | i64 | `ReduceMax` MergeMode 的 block 级版本 | 两值取大 |
| `UAbs` | i64 | i64 | `abs.s`（FP） | 绝对值 |
| `UAssert` | i64 (cond) | — | `TEQ`/`TNE` 陷阱指令 | cond 为 0 时 panic，用于前置条件 |

> **设计说明**：MIPS 没有独立的 min/max 指令，需要用 `SLT` + `MOVZ` 两步实现。
> URP 数据流模型中合并成一个 opcode 更自然，减少 block 数量。

---

## 四、PayloadValue 类型扩展

当前只有 `I64` 和 `Str`。第二批指令基本够用，但若要对标 MIPS 浮点协处理器（COP1），
需要增加 `F64` 类型，届时再规划 `FAdd`/`FSub`/`FMul`/`FDiv` 等浮点 opcode。

```
PayloadValue::F64(f64)   ← 对应 MIPS COP1 单精度/双精度
```

GPU executor 对浮点 opcode 的支持成本极低（WGSL 原生支持 f32）。

---

## 五、实现顺序建议

```
第一批（算术+比较）  →  第二批（逻辑+移位）  →  第三批（字符串）  →  第四批（条件选择+聚合）
```

每批完成后：
1. 在 `src/ir.rs` 的 `Opcode` enum 新增变体
2. 在 `src/executor.rs` 的 `eval_opcode` 新增 match 分支（CPU 路径）
3. 在 `src/gpu_executor.rs` 新增 GPU 路径（第一批算术指令优先 GPU）
4. 补充对应测试

## 六、设计原则总结（借鉴 MIPS）

| MIPS 原则 | URP 对应做法 |
|-----------|-------------|
| 简洁统一（固定格式） | 每个 opcode 输入数固定，不搞可变参数指令 |
| Load/Store 分离 | URP 无内存层，PayloadValue 即"寄存器"，天然分离 |
| 正交性（任意寄存器可用） | 任意 block 输出均可作为任意 block 的 inputs，无约束 |
| 常见情况快速 | 常量直接用 `UConstI64`，不需要 LUI+ORI 两步 |
| 无延迟槽 | URP DAG 拓扑调度无此问题，天然避免 MIPS 历史包袱 |
| 协处理器模型 | GPU executor 对应 COP1，未来 FPGA executor 对应 COP2 |
