## 项目背景

### URP核心概念

URP不是传统意义上的虚拟CPU管理器，而是一个全新的执行架构：

- **节点宇宙观**: CPU、GPU、USB设备、内存、网络、规则、结构都被视为可调度的算力节点
- **图执行模型**: 任务被表示为IRGraph（中间表示图），经过融合、分区、调度后分发执行
- **统一调度语义**: 通过PartitionDAGScheduler实现DAG拓扑调度和AsyncLane并行执行

### URP架构流程

```
IRGraph → fusion → partition → policy-based binding
        → PartitionDAGScheduler → execute (CPU/GPU/USB)
        → packet route → reducer merge
```

### 关键特性

| 特性 | 说明 |
|------|------|
| **IRGraph执行模型** | 任务先被结构化为有向图，再执行 |
| **Block Fusion** | 自动融合相邻可合并块 |
| **Graph Partition** | 按资源需求分区 |
| **Policy-based Scheduling** | 可替换调度策略（Multifactor/Round-Robin等） |
| **PartitionDAGScheduler** | 完整DAG拓扑调度，AsyncLane并行执行分区 |
| **ET-WCN Cooling** | 基于熵温度冷却的分区绑定优化器 |
| **Topology-aware Cost Model** | 考虑区域、带宽、惯性的成本模型 |
| **IRGraph JSON序列化** | serde_json双向序列化 |
| **ZeroCopyContext** | SharedMemoryRegion + BufferPool + InertiaBufferCache |
| **GPU/USB执行器** | 支持GPU着色器和USB CDC设备执行 |

## 测试方案设计

### 重要概念：物理核心 vs 虚拟节点

URP测试中的"核"需要区分两个概念：

| 概念 | 说明 | 本测试环境 |
|------|------|-----------|
| **物理CPU核心** | 实际的处理器核心数 | **100个** (Intel Xeon Gold 6230R) |
| **URP虚拟节点** | URP软件抽象的调度节点 | 100/200/500个（可任意创建） |

**URP节点模型特点**：
- 节点是软件抽象，不等于物理硬件
- 可以创建任意数量的虚拟节点来模拟大规模集群
- 虚拟节点最终会映射到物理核心上执行
- 多个虚拟节点可能竞争同一个物理核心

**测试场景说明**：
- **100核测试**: 创建100个虚拟节点，与物理核心数1:1对应
- **200核测试**: 创建200个虚拟节点，模拟超配场景
- **500核测试**: 创建500个虚拟节点，测试调度器极限能力

这种设计允许在单机上模拟和测试大规模集群调度场景，无需实际部署数百台物理机器。

### 100核调度测试目标

验证URP在以下场景下的调度能力：

1. **大规模节点创建** - 100个CPU核心的节点建模
2. **DAG并行调度** - 独立管道的并行执行
3. **分区级绑定** - 图分区到节点的映射
4. **惯性亲和** - 基于缓存局部性的调度优化
5. **性能扩展性** - 不同规模下的吞吐量表现

### 节点模型设计

```rust
let nodes: Vec<Node> = (0..100).map(|i| {
    Node::new(
        format!("cpu_core_{}", i),
        NodeType::Cpu,
        100.0  // compute_capacity
    )
    .with_zone("zone-a")  // 分散到多个zone
    .with_host_id("cluster_host")
    .with_tag("cpu")
}).collect();
```

### 节点属性说明

| 属性 | 类型 | 说明 |
|------|------|------|
| `node_id` | String | 节点唯一标识 |
| `node_type` | NodeType | Cpu/Gpu/Qcu/Memory/Network/Rule/Structure/Usb |
| `host_id` | String | 宿主机标识，同host走本地ring，不同host走远程包 |
| `zone` | String | 区域标识，影响路由开销 |
| `tags` | Vec<String> | 能力标签，如["cpu", "gpu"] |
| `compute_capacity` | f32 | 算力容量，越大得分越高 |
| `bandwidth` | f32 | 带宽，影响路由开销计算 |
| `inertia_keys` | Vec<String> | 已缓存的惯性键，命中时调度优先度大幅提升 |

### 硬件性能分析

基于测试环境的硬件配置，进行性能分析：

**CPU利用率分析**:
- 100核测试中，理论上可达到100%并行度
- 实际执行受以下因素影响：
  - **NUMA架构**: 2个NUMA节点，跨节点访问有延迟
  - **L3缓存共享**: 36MB L3缓存由50个物理核心共享
  - **超线程**: Intel Xeon Gold 6230R支持超线程，逻辑核心=物理核心×2

**内存带宽分析**:
- 78GB内存容量充足，测试未触及内存瓶颈
- 72GB可用内存表明系统负载较轻
- 交换分区使用616MB表明无内存压力

**优化后实际性能vs理论性能**:
```
理论峰值: 100核心 × 2.1GHz = 210 GHz总计算能力
优化前吞吐: 8964 ops/sec (约每核90 ops/sec)
优化后吞吐: 10,250 ops/sec (约每核103 ops/sec)

单操作平均时间: 约0.10ms (优化前0.11ms)
```

**网络性能影响**:
- 测试环境网络下载速度约2.8 MB/s
- 本地测试主要使用本地ring通道，网络影响较小
- 在跨host场景下，网络带宽会成为瓶颈

**超配场景分析**:

| 超配比 | 虚拟节点 | 物理核心 | 优化前 | 优化后 | 提升幅度 |
|--------|----------|----------|--------|--------|----------|
| 1:1 | 100 | 100 | 8964 ops/sec | ~10,250 ops/sec | +14.3% |
| 2:1 | 200 | 100 | 6608 ops/sec | ~6,913 ops/sec | +4.6% |
| 5:1 | 500 | 100 | 3380 ops/sec | ~4,089 ops/sec | +21.0% |

**500核提升最显著的原因**：执行路径优化中，Block查找从O(n)降到O(1)、Edge查找从O(E)降到O(1)，这些优化对大规模场景效果最明显。

**实际应用建议**：
- 生产环境建议超配比不超过2:1
- 对于计算密集型任务，建议1:1或更低的超配比
- 对于I/O密集型任务，可以适当提高超配比

## 技术亮点

### 1. DAG拓扑调度

PartitionDAGScheduler使用Kahn算法实现DAG拓扑排序：

```rust
// 计算分区入度
let mut in_degree = HashMap::new();
for (pid, _) in &partitions {
    in_degree.insert(pid.clone(), 0);
}
for edge in &dag_edges {
    *in_degree.get_mut(&edge.dst).unwrap() += 1;
}

// 初始入度为0的分区可执行
let mut ready: Vec<_> = in_degree.iter()
    .filter(|(_, &d)| d == 0)
    .map(|(pid, _)| pid.clone())
    .collect();
```

### 2. AsyncLane并行执行

使用信号量控制并发度，实现真正的并行执行：

```rust
pub struct AsyncLane<F, Fut>
where
    F: Fn(Partition) -> Fut,
    Fut: Future<Output = Vec<BlockExecutionResult>> + Send,
{
    max_concurrent: usize,
    execute_fn: F,
    _phantom: PhantomData<Fut>,
}
```

### 3. 多因素调度策略

MultifactorPolicy综合考虑多个因素：

```rust
score = 0.0
if node.has_tag(required_tag):
    score += 2.0
if node.zone == preferred_zone:
    score += 1.5
score += 0.1 * node.compute_capacity
score += 0.02 * node.bandwidth
if node.has_inertia_key(inertia_key):
    score += 3.0  // 惯性加成权重最高
```

**优先级**: 惯性命中 (3.0) > 标签匹配 (2.0) > 区域匹配 (1.5) > 容量 (0.1x) > 带宽 (0.02x)

### 4. 本地Ring路由优化

同host节点优先走本地ring通道，避免网络开销：

```rust
if src.host_id == dst.host_id {
    // LocalRingTunnel - 内存环形队列，无网络开销
    cost = 0.1
} else {
    // RemotePacketLink - TCP网络传输
    cost = 10.0 + zone_cost + bandwidth_cost
}
```
